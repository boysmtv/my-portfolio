# Web Portfolio Visual Evolution Brief

## Stack Directive
- Required stack: `web-nextjs`
- Required platform: web
- Required baseline: evolve the existing Next.js App Router project in place
- Do not switch to another stack or template family
- Preserve the current project as the source baseline and improve it visually


## Context
This project is an existing personal portfolio for Dedy Wijaya, built with Next.js. The current product already renders and builds successfully, but the visual presentation still feels too generic and template-like for a senior engineering leader portfolio.

This brief is intentionally focused on visual quality, storytelling, information hierarchy, and portfolio credibility. Technology changes are not the goal unless a very small technical adjustment is required to unlock the visual solution.

## Primary Goal
Transform the current portfolio into a high-trust, visually distinctive, senior-level engineering portfolio that feels:
- credible
- memorable
- premium
- evidence-driven
- clearly personal to Dedy Wijaya

The portfolio must stop feeling like a polished generic template and start feeling like the portfolio of a technical lead / senior engineer with real delivery experience in fintech, payments, Android, and backend systems.

## Non-Goals
Do not spend the main effort on:
- replacing the framework
- migrating the tech stack
- backend platform work
- CMS integration
- heavy infrastructure work
- broad technical refactors that do not improve presentation

Minor fixes are allowed if they are needed to support a better visual result.

## Current Problems To Solve
### 1. Hero is not strong enough
The hero section is visually clean, but it still feels like a startup landing page hero rather than an executive-level engineering profile.

Problems:
- not enough proof at first glance
- not enough hard signals of credibility
- positioning is still too generic
- CTA is weak

### 2. Projects feel decorative instead of authoritative
The projects section is currently attractive, but it behaves like a gallery of similar cards instead of a curated portfolio of high-value case studies.

Problems:
- too many projects have equal visual weight
- there is no clear hierarchy between flagship projects and supporting projects
- project cards look polished but not evidence-driven
- placeholder/random images reduce trust

### 3. Case study depth is too shallow
Project detail presentation currently looks like an expanded modal, but does not yet deliver the full narrative expected from senior-level work.

Problems:
- challenge is not told deeply enough
- personal responsibility is not explicit enough
- decisions and trade-offs are not visible enough
- impact is not emphasized enough

### 4. Visual identity is still too generic
The dark modern style is decent, but it still looks like a familiar premium template instead of a unique engineering-fintech identity.

Problems:
- atmosphere is not distinctive enough
- color hierarchy is safe but not memorable
- visual language is not yet strongly tied to systems, payments, scale, fintech, architecture, or platform thinking

### 5. Page rhythm is too repetitive
Many sections use a similar title-paragraph-card pattern, which makes the page feel long and visually predictable.

Problems:
- too many sections follow the same rhythm
- not enough contrast between major narrative moments
- not enough variation in density and visual emphasis

### 6. CTA and portfolio trust signals are weak
The portfolio needs stronger trust and conversion surfaces.

Problems:
- CTA does not feel intentional enough
- some interactions feel decorative rather than outcome-focused
- the portfolio should show faster signals of seniority, scale, ownership, and impact

## Product Direction
This portfolio should feel like:
- a high-trust engineering profile
- a systems-minded fintech portfolio
- a curated leadership portfolio
- a portfolio that communicates architecture, delivery, scale, ownership, and results

It should not feel like:
- a generic portfolio starter
- a UI showcase without proof
- a landing page clone
- a random set of polished cards

## Visual Direction
Adopt a stronger engineering-fintech visual identity.
Possible cues include:
- control-room energy
- systems-dashboard structure
- architecture blueprint accents
- premium terminal / operations tone
- restrained but strong signal colors

Do not make it noisy or gimmicky.
The result should still feel mature, clean, and executive.

## Content and Hierarchy Requirements
### Hero
Improve hero so it immediately communicates:
- who Dedy is
- seniority
- domain strength
- strongest proof points
- why this portfolio matters

Hero should include fast credibility markers such as:
- years of experience
- fintech / payments specialization
- Android + backend scope
- delivery / systems / architecture signals

### Projects
Restructure project presentation so it has hierarchy:
- 2 or 3 flagship case studies with strong visual emphasis
- remaining projects as supporting evidence

Do not give equal weight to every project.
The portfolio must clearly show which projects are the strongest proof.

### Case Study Presentation
Each featured project should communicate:
- context
- business problem
- role / ownership
- architecture decisions
- constraints
- key challenges
- outcomes / impact

The presentation should feel like a real case study, not just a pretty popup.

### Information Compression
Reduce repetitive sections or make them more distinct.
If multiple sections overlap in meaning, merge or differentiate them visually and structurally.

### Trust Signals
Surface proof more aggressively through UI, such as:
- scale
- transaction / uptime / reliability cues
- architecture decisions
- responsibility markers
- role clarity
- delivery outcomes

## UX Requirements
- Keep the site responsive and polished on desktop and mobile.
- Preserve smooth interactions, but avoid motion that feels ornamental without purpose.
- Interaction should support credibility and navigation, not distract from content.
- Modal/detail experiences should feel premium and intentional.
- Calls to action should be clear and useful.

## Constraints
- Evolve the existing project, do not throw it away without reason.
- Keep the implementation grounded in the current Next.js project unless a small adjustment is required.
- Prioritize visual improvement and storytelling improvement over technical novelty.
- Avoid introducing unnecessary complexity.

## Expected Output
The resulting portfolio should:
- look noticeably more distinctive
- feel more senior and more credible
- create stronger hierarchy across hero, featured work, and supporting sections
- communicate real engineering leadership more clearly
- reduce the generic-template feeling

## Acceptance Criteria
The work is successful if:
1. The hero feels stronger, more specific, and more authoritative.
2. The portfolio no longer feels like a generic premium template.
3. Flagship projects clearly stand out from supporting projects.
4. Project storytelling shows role, impact, and decision quality better.
5. Visual identity feels more intentional and more tied to engineering/fintech.
6. Page rhythm becomes more varied and less repetitive.
7. The overall experience remains polished and responsive.
8. The existing project is evolved, not replaced casually.

## Delivery Instruction For AI-Agent
Use this project as the existing baseline.
Do a visual and UX evolution pass with strong judgment.
Do not blindly preserve current layout decisions if they are weakening the portfolio.
Improve the design as a product-facing engineering portfolio, not as a generic frontend demo.
